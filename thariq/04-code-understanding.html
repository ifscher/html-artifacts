Como a autenticação flui pelo codebase

O Birchline usa sessões baseadas em cookie: o browser nunca guarda um bearer token diretamente. Toda requisição autenticada bate em /api/*, passa por um único middleware verifyToken() e resolve para uma linha Session que os handlers downstream leem de req.ctx. O middleware é o único lugar que fala com o session store, que é o único lugar que fala com a tabela sessions — então há exatamente um trust boundary para raciocinar.

export function AuthProvider({ children }: Props) {
  const [user, setUser] = useState<User | null>(null);

  useEffect(() => {
    fetch('/api/session', { credentials: 'include' })
      .then(r => r.ok ? r.json() : null)
      .then(setUser);
  }, []);

  return <AuthCtx.Provider value={{ user }}>{children}</AuthCtx.Provider>;
}

src/middleware/auth.ts :14-31

Este é o trust boundary. verifyToken lê o cookie assinado fw_sid, pede ao SessionStore para resolvê-lo, e ou popula req.ctx.session ou responde 401. Todas as rotas protegidas do app são montadas atrás desta função, então alterar seu comportamento muda a autenticação globalmente.

ver código 
// src/middleware/auth.ts
export async function verifyToken(req, res, next) {
  const raw = req.signedCookies['fw_sid'];
  if (!raw) return res.status(401).end();

  const session = await SessionStore.get(raw);
  if (!session || session.expiresAt < Date.now()) {
    return res.status(401).end();
  }

  req.ctx = { session };
  next();
}

src/lib/sessionStore.ts :8-52

SessionStore é um cache read-through pequeno: verifica primeiro um LRU em processo, depois recorre ao Postgres. Escritas (create, revoke) sempre vão direto ao banco e invalidam a entrada no cache para que outros workers não sirvam uma sessão obsoleta.

ver código 
// src/lib/sessionStore.ts
const cache = new LRU<string, Session>({ max: 5000, ttl: 60_000 });

export const SessionStore = {
  async get(id: string) {
    const hit = cache.get(id);
    if (hit) return hit;
    const row = await db.one(SELECT_SESSION, [id]);
    if (row) cache.set(id, row);
    return row ?? null;
  },
  /* create, revoke, touch ... */
};

db/migrations/004_sessions.sql :1-18

A tabela sessions é indexada por um id aleatório de 32 bytes (o valor do cookie) com um índice de cobertura em user_id para "deslogar em todos os dispositivos". A expiração é verificada tanto aqui (expires_at) quanto no middleware como defesa em profundidade.

ver código 
-- db/migrations/004_sessions.sql
create table sessions (
  id          text primary key,
  user_id     uuid not null references users(id),
  created_at  timestamptz default now(),
  expires_at  timestamptz not null,
  ip          inet,
  user_agent  text
);
create index sessions_user_id_idx on sessions(user_id);

Arquivos-chave

  • src/middleware/auth.ts Único ponto de entrada para autenticação de requisições.
  • src/lib/sessionStore.ts Lookup de sessão via LRU + Postgres; único chamador do banco.
  • src/server/routes/session.ts Retorna a sessão atual para o cliente.
  • src/server/routes/login.ts Troca credenciais por cookie via SessionStore.create.
  • src/app/providers/AuthProvider.tsx Contexto client-side que espelha a sessão do servidor.
  • db/migrations/004_sessions.sql Schema da tabela sessions e índices.

Atenção

  • O LRU no SessionStore é por processo. Revogar uma sessão só limpa o cache local — outros workers podem servi-la por até 60s até o TTL expirar.
  • verifyToken compara expiresAt com Date.now(), mas a coluna é timestamptz. O driver retorna um Date, então a comparação funciona, mas não refatore para string pura sem ajustar a verificação.
