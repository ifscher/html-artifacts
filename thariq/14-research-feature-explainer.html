Como funciona o rate limiting no birchline/api

TL;DR — Cada requisição passa pelo middleware rateLimit(), que resolve o chamador para uma bucket key, busca um token-bucket no Redis e consome um token ou retorna 429. Os limites são declarados por rota em config/limits.yaml; rotas sem entrada herdam o tier default (100 req/min por chave de API).

O caminho da requisição, passo a passo

Expanda cada etapa para ver o que executa e onde está no código. O caminho completo tem ~40 linhas e adiciona cerca de 0,4 ms p50 a cada requisição.

1 · Identificar o chamador middleware/ratelimit.ts:21

O middleware primeiro reduz a requisição a uma bucketKey: chave de API se um cabeçalho Authorization estiver presente, caso contrário o IP do cliente (via cadeia x-forwarded-for, confiando apenas no nosso próprio LB). Tráfego IP anônimo recebe um tier padrão muito mais baixo.

2 · Buscar o bucket lib/tokenBucket.ts:9

O nome da rota mais a bucket key mapeiam para um hash Redis (rl:{route}:{key}) contendo tokens e updatedAt. Se a chave não existir, ela é criada sob demanda com capacidade total — sem período de aquecimento.

3 · Recarregar e consumir lib/tokenBucket.ts:31

O recarregamento é calculado a partir do tempo decorrido (rate × Δt, limitado ao burst), depois um token é subtraído. Toda a operação read-modify-write roda como um único script Lua para que requisições concorrentes não gastem tokens em duplicidade.

4 · Rejeitar quando vazio middleware/ratelimit.ts:48

Se o script retornar tokens < 0, o middleware interrompe com 429 Too Many Requests e define Retry-After com os segundos até um token ser recarregado. Respostas bem-sucedidas sempre incluem X-RateLimit-Remaining.

Configurar um limite na sua rota

Você não precisa tocar no middleware. Adicione uma entrada em config/limits.yaml com o nome da rota como chave e, opcionalmente, marque a rota para que o middleware a encontre.

# config/limits.yaml
default:
  rate: 100/min
  burst: 120

search.query:
  rate: 20/min
  burst: 40
  key: api_key        # ou: ip
// routes/search.ts
router.post(
  "/search",
  rateLimit("search.query"),
  handler,
);
HTTP/1.1 429 Too Many Requests
Retry-After: 17
X-RateLimit-Limit: 20
X-RateLimit-Remaining: 0

{ "error": "rate_limited", "retry_after": 17 }
★
Se você precisar apenas do tier padrão, não é necessário criar uma entrada no YAML — basta envolver o handler em rateLimit() sem argumento. O nome da rota é inferido a partir do caminho.

Atenção

  • Os limites são por processo em desenvolvimento. O cliente Redis usa um mapa em memória quando REDIS_URL não está definido, então testes locais não refletem o comportamento real do cluster.
  • Burst ≠ rate. burst é a capacidade do bucket; um chamador ocioso por um minuto pode disparar burst requisições de uma vez mesmo que rate seja baixo.
  • Respostas em streaming contam uma vez. O token é consumido no início da requisição; um stream SSE de 30 segundos ainda custa um token.

FAQ

Como isentar tráfego interno?
Defina x-birchline-internal: 1 no chamador; o middleware verifica isso contra o nome do peer mTLS e ignora o bucket completamente.
Onde vejo quem está sendo limitado?
Cada 429 emite uma métrica ratelimit.rejected marcada com rota e tipo de chave. Há um painel no Grafana em API → Saúde.
Um único usuário pode ter um limite maior?
Sim — adicione a chave de API dele em overrides: no YAML. Os overrides são recarregados sem necessidade de deploy.
