Três formas de implementar busca com debounce

Prompt Mostre três formas diferentes de implementar busca com debounce para o input de filtro de tarefas no nosso codebase React, com os trade-offs de cada uma.

01useEffect + setTimeout inline

A lógica de debounce fica diretamente dentro do componente que possui o input.

export function TaskSearch() {
  const [draft, setDraft] = useState('');
  const [query, setQuery] = useState('');

  useEffect(() => {
    const id = setTimeout(() => setQuery(draft), 300);
    return () => clearTimeout(id);
  }, [draft]);

  const { data } = useTasks({ search: query });

  return (
    <input
      value={draft}
      onChange={(e) => setDraft(e.target.value)}
      placeholder="Filtrar tarefas…"
    />
  );
}
Prós
Contras
Nenhuma abstração nova para aprender
Lógica duplicada em todo lugar que tem busca
Fácil de depurar no devtools
Dois estados para um único valor conceitual
Sem nova dependência ou mudança no bundle
Constante de delay enterrada no corpo do componente
Impacto no bundle: +0 kb Testabilidade: média Reuso: baixo Seguro para SSR: sim

02Hook customizado useDebounce

Extrai o timer para um hook compartilhado em src/hooks/.

// src/hooks/useDebounce.ts
export function useDebounce<T>(value: T, ms = 300): T {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const id = setTimeout(() => setDebounced(value), ms);
    return () => clearTimeout(id);
  }, [value, ms]);
  return debounced;
}

// TaskSearch.tsx
const [draft, setDraft] = useState('');
const query = useDebounce(draft, 300);
const { data } = useTasks({ search: query });
Prós
Contras
Um único import reutilizado no filtro, command bar e busca do board
Mais um arquivo para manter e documentar
Trivial de testar com fake timers
O genérico T esconde um pouco a intenção
O delay é um argumento visível e ajustável
Ainda re-renderiza a cada tecla pressionada
Impacto no bundle: +0,2 kb Testabilidade: alta Reuso: alto Seguro para SSR: sim

03Biblioteca externa pequena

Adota use-debounce para valores e callbacks.

import { useDebouncedCallback }
  from 'use-debounce';

export function TaskSearch() {
  const [query, setQuery] = useState('');

  const onChange = useDebouncedCallback(
    (next: string) => setQuery(next),
    300,
    { leading: false, maxWait: 1000 },
  );

  const { data } = useTasks({ search: query });

  return (
    <input
      defaultValue=""
      onChange={(e) => onChange(e.target.value)}
    />
  );
}
Prós
Contras
leading / trailing / maxWait tratados automaticamente
Nova dependência de runtime para auditar e atualizar
Forma de callback evita re-renders intermediários
Input não controlado diverge dos padrões de formulário do Birchline
Casos de borda bem testados (unmount, flush, cancel)
~1,4 kb gzipped para algo que poderíamos implementar
Impacto no bundle: +1,4 kb Testabilidade: alta Reuso: alto Seguro para SSR: sim

Recomendação

Use a abordagem 02, o hook customizado useDebounce. O Birchline já tem três lugares que replicam o padrão inline (filtro de tarefas, command palette, seletor de membros), então extrair um hook compartilhado elimina a duplicação sem adicionar nova dependência.

Reavalie a abordagem 03 apenas se precisarmos de semânticas de maxWait ou flush() — a biblioteca justifica seu custo no bundle quando os requisitos ultrapassam um hook de dez linhas.
